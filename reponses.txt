
Q1 Expliquez le rôle de Vite dans un projet Vue. À quoi servent main.js, App.vue, le dossier components et le dossier views ?

Vite, c'est l'outil qui fait tourner le projet. En développement, il lance un serveur local avec du rechargement instantané : dès qu'on modifie un fichier, la page se met à jour toute seule sans tout recharger. Et quand on veut mettre en production, c'est lui qui build le projet (il regroupe et optimise tout le code) pour avoir des fichiers prêts à déployer.

Pour les fichiers :
- main.js, c'est le point d'entrée de l'appli. C'est là qu'on crée l'application Vue, qu'on la branche sur le HTML, et qu'on branche aussi les plugins comme le router et Pinia.
- App.vue, c'est le composant racine, celui qui englobe tout le reste. En général il contient la navbar et un endroit où les pages viennent s'afficher (le router-view).
- Le dossier components, c'est là qu'on met les petits bouts réutilisables : une carte, un bouton, une barre de recherche... des morceaux qu'on réutilise à plusieurs endroits.
- Le dossier views, c'est les pages complètes, celles reliées à une route. Par exemple la page d'accueil, la page détail d'un événement, la page de création. Une view utilise souvent plusieurs components à l'intérieur.


Q2 Expliquez le rôle de v-bind, v-model, v-if, v-for et @click. Donnez un court exemple d'utilisation pour au moins trois de ces directives.

- v-bind sert à relier un attribut HTML à une donnée. Au lieu d'écrire une valeur en dur, on la branche sur une variable. On peut l'écrire en raccourci avec juste ":".
- v-model, c'est la liaison dans les deux sens sur un champ de formulaire : la variable met à jour l'input, et l'input met à jour la variable. Pratique pour une barre de recherche ou un formulaire.
- v-if affiche ou pas un élément selon une condition. Utile pour gérer les états vide / chargement / erreur.
- v-for permet de boucler sur une liste pour afficher un élément par entrée.
- @click écoute le clic sur un élément et déclenche une fonction.

Exemples :

<!-- v-model : le champ et la variable search restent synchronisés -->
<input v-model="search" placeholder="Rechercher un événement" />

<!-- v-for : on affiche une carte par événement -->
<EventCard v-for="event in events" :key="event.id" :event="event" />

<!-- v-if + @click -->
<p v-if="events.length === 0">Aucun événement trouvé</p>
<button @click="loadEvents">Recharger</button>

Ici on voit aussi v-bind avec ":key" et ":event" qui passent des valeurs dynamiques.


Q3 Expliquez l'intérêt de la Composition API avec <script setup>. Quelle est la différence entre ref() et computed() ? Donnez un exemple concret.

La Composition API avec <script setup>, ça permet d'écrire la logique d'un composant de façon plus directe et plus lisible. On déclare ses variables et ses fonctions à la suite, sans passer par une grosse structure avec data, methods, computed séparés. Et surtout, quand un composant devient gros, on peut regrouper le code par fonctionnalité au lieu de l'éparpiller, et même sortir de la logique dans des fichiers à part pour la réutiliser.

La différence entre ref() et computed() :
- ref() sert à créer une donnée réactive qu'on peut modifier nous-mêmes. Par exemple un compteur, un texte de recherche, une liste. On lit et on change sa valeur avec .value.
- computed() sert à créer une valeur calculée à partir d'autres données. On ne la modifie pas directement : elle se recalcule toute seule quand ce dont elle dépend change. En plus elle est mise en cache, donc elle ne recalcule que si nécessaire.

Exemple concret : une recherche qui filtre une liste d'événements.

const search = ref('')
const events = ref([...]) // la liste complète

const filteredEvents = computed(() =>
  events.value.filter(e =>
    e.title.toLowerCase().includes(search.value.toLowerCase())
  )
)

Ici search et events sont des ref (des données qu'on modifie). filteredEvents est un computed : dès que search change, la liste filtrée se met à jour automatiquement, sans qu'on ait à la recalculer à la main.


Q4 Expliquez comment les composants communiquent avec props, emits et slots. Expliquez aussi le rôle de Vue Router et de Pinia dans une application composée de plusieurs pages.

Pour la communication entre composants :
- Les props, c'est ce qu'un composant parent envoie à un composant enfant. Ça descend du parent vers l'enfant. Par exemple la page passe un objet "event" à une EventCard pour qu'elle sache quoi afficher.
- Les emits, c'est l'inverse : l'enfant prévient le parent qu'il s'est passé quelque chose. Par exemple un EventForm qui émet un événement "submit" avec les données du formulaire, et c'est le parent qui décide quoi en faire.
- Les slots, ça permet au parent d'injecter du contenu à l'intérieur d'un composant. C'est pratique pour un composant réutilisable genre BaseCard ou BaseModal : le composant gère l'apparence (la bordure, le fond...) et le parent met le contenu qu'il veut dedans.

Vue Router, c'est ce qui gère la navigation entre les pages. Comme on est sur une application d'une seule page, on ne recharge pas vraiment le navigateur : le router regarde l'URL et affiche la bonne view. Par exemple "/" montre le dashboard, "/events/:id" le détail d'un événement, "/create" le formulaire. Il gère aussi les liens et le passage de paramètres dans l'URL (l'id de l'événement par exemple).

Pinia, c'est le store, l'endroit où on centralise les données partagées entre plusieurs pages. Plutôt que de faire passer les mêmes infos de composant en composant, on les met dans le store et n'importe quelle page peut y accéder. Par exemple la liste des événements peut être stockée dans un store : le dashboard l'affiche, la page détail y récupère un événement précis, et quand on en crée un nouveau on l'ajoute au store. Ça évite de recharger les mêmes données partout et de galérer à les faire circuler.
